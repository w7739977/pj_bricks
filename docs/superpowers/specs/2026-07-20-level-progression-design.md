# 田园连连看关卡进阶系统设计

日期：2026-07-20
状态：已确认

## 1. 背景

当前游戏只有单局随机模式。顶部的“第 N 局”只是当前页面会话中的重开次数；清空棋盘后出现“游戏胜利”弹窗，玩家只能重新生成一局。刷新页面后所有进度丢失。

本次改造将游戏调整为持续关卡模式：清空棋盘即完成当前关，经过结算弹窗进入下一关；前五关具有稳定、可复现的盘面，后续关卡持续随机；难度通过图案种类增加以及提示、重排资源减少来提升。

## 2. 目标

- 第 1～5 关使用固定随机种子，重试时恢复同一盘面。
- 第 6 关起使用随机种子；同一关在通关前始终保持同一盘面。
- 第 1 关使用现有 14 种图案，之后每关加入一种新水果，第 7 关达到 20 种。
- 新增香蕉、橙子、梨、樱桃、桃子、西瓜六种手绘 SVG 图案。
- 棋盘继续保持 14 行 × 10 列、共 140 个方块。
- 提示和重排次数随关卡减少。
- 手动重排和死局救援共用同一个重排资源池。
- 清盘后显示结算弹窗，再进入下一关。
- “重开”改为“重试当前关”。
- 使用 `localStorage` 保存当前关卡、最高已解锁关卡及随机关卡种子。
- 保持现有点击消除、固定推链拖拽、动态 pitch、动画所有权、无障碍和 reduced-motion 行为。

## 3. 非目标

首版不包含：

- 账号、云端存档或跨设备同步；
- 关卡选择界面；
- 排行榜、积分、星级、连击奖励；
- 时间限制或步数限制；
- 障碍物、锁定格或特殊方块；
- 不同棋盘尺寸或异形棋盘；
- 保证盘面存在完整清盘路径；
- 保存清到一半的棋盘状态。

## 4. 关卡形态

### 4.1 固定与随机关卡

- 第 1～5 关：关卡配置中保存固定种子。
- 第 6 关起：首次解锁下一关时，通过 `crypto.getRandomValues()` 生成种子并立即保存。
- 同一关的重试、刷新和重新打开都使用相同种子，因此恢复同一初始盘面。
- 完成关卡时为下一关生成新种子；进入下一关后才开始使用它。
- 关卡没有最终上限，第 7 关以后持续生成使用全部 20 种图案的随机关卡。

随机盘面沿用现有策略：最多尝试 1000 次，保证初始盘面至少存在一对可消除方块，但不保证存在完整清盘路径。死局是玩法的一部分，由有限重排或重试处理。

### 4.2 图案解锁

| 关卡 | 图案数量 | 新解锁图案 |
|---|---:|---|
| 1 | 14 | 现有全部图案 |
| 2 | 15 | 香蕉 |
| 3 | 16 | 橙子 |
| 4 | 17 | 梨 |
| 5 | 18 | 樱桃 |
| 6 | 19 | 桃子 |
| 7+ | 20 | 西瓜 |

新增图案沿用现有视觉规范：64×64 `viewBox`、深棕色粗描边、圆角线帽和连接、明亮平涂、少量内部纹理，并保持小尺寸下轮廓可辨。

### 4.3 资源难度曲线

| 关卡 | 提示次数 | 重排次数 |
|---|---:|---:|
| 1 | 3 | 3 |
| 2 | 3 | 2 |
| 3 | 2 | 2 |
| 4 | 2 | 1 |
| 5 | 1 | 1 |
| 6 | 1 | 1 |
| 7+ | 0 | 1 |

规则：

- 成功显示一对提示后才扣除提示次数。
- 当前盘面已经死局时，点击提示不扣次数，直接进入死局处理。
- 手动点击重排和死局弹窗中的救援重排共用同一计数。
- 重试当前关后，提示和重排次数恢复为该关初始值。
- 次数为零的工具按钮保持可见但禁用，并通过 ARIA 状态说明不可用原因。

## 5. 模块设计

### 5.1 `game/levels.js`

新增 DOM-free 关卡规则模块，负责：

- 保存第 1～5 关固定种子；
- 返回关卡图案数量、图案列表、提示次数和重排次数；
- 创建稳定的确定性伪随机数生成器；
- 规范化 32 位整数种子；
- 提供关卡解锁图案信息，供结算弹窗使用。

接口：

```js
getLevelConfig(levelNumber, { randomSeed = null } = {})
createSeededRng(seed)
createRandomSeed(cryptoSource)
```

`getLevelConfig()` 返回不可变配置数据，不访问 DOM、`localStorage` 或交互状态。第 1～5 关一律忽略调用者传入的 `randomSeed`，使用配置中的固定种子；第 6 关以后必须传入已规范化的无符号 32 位整数种子，否则抛出 `RangeError`。测试必须覆盖向固定关传入不同外部种子时配置和盘面仍保持一致。

### 5.2 `game/progress.js`

新增存档模块，只负责进度的校验、读取和写入。固定存储键为 `lianliankan.progress.v1`，存档格式：

```js
{
  version: 1,
  currentLevel: 6,
  highestUnlocked: 6,
  levelSeed: 123456789
}
```

规则：

- 第 1～5 关使用固定种子，`levelSeed` 可以为 `null`。
- 第 6 关以后必须保存有效的无符号 32 位整数种子。
- `currentLevel` 与 `highestUnlocked` 必须为正整数。
- 首版没有关卡选择或回到旧关入口，正常存档必须满足 `currentLevel === highestUnlocked`；保留 `highestUnlocked` 字段是为未来选关能力预留。
- 清盘时将两者同时更新为下一关；因此玩家在结算弹窗期间关闭或刷新，也会从下一关继续。
- 不保存当前棋盘、剩余资源或已消除位置；刷新等同于重试当前关。

接口：

```js
createDefaultProgress()
parseProgress(rawValue)
loadProgress(storage)
saveProgress(storage, progress)
```

`loadProgress()` 和 `saveProgress()` 返回 `{ ok: true, value }` 或 `{ ok: false, error }`。存储读写失败不阻止游戏继续使用内存状态。

### 5.3 `game/board.js`

将当前固定的 `KINDS = 14`、`PER_KIND = 10` 生成方式改为参数化生成：

```js
createBoard({ iconIndices, rng })
// => { ok: true, board } | { ok: false, reason: 'no-solvable-pair' }
```

`iconIndices` 是 `ICON_NAMES` 中图案位置的整数数组；棋盘继续使用现有 `(number | null)[][]` 数据模型，因此 `interaction.js` 仍可通过 `ICON_NAMES[value]` 渲染，不把字符串名称写入棋盘。棋盘尺寸仍固定为 `ROWS = 14`、`COLS = 10`。

生成算法：

1. 将 140 个方块视为 70 对。
2. 按轮转方式把 70 对分配给当前已解锁图案。
3. 每种图案至少获得一对，且各图案数量差不超过两个方块。
4. 展开成 140 个图案编号后，使用传入的确定性 RNG 进行 Fisher–Yates 洗牌。
5. 如果初始盘面没有任何可消除对，则继续使用同一 RNG 流重新洗牌，最多尝试 1000 次。

结果保证：

- 总数始终为 140；
- 每种图案数量均为偶数；
- 所有已解锁图案至少出现一对；
- 相同配置和种子得到相同盘面；
- 初始盘面至少有一对可消除方块。

`reshuffleInPlace(board, rng)` 同样返回 `{ ok: true } | { ok: false, reason: 'no-solvable-pair' }`。它在开始前保留剩余图案快照；1000 次内无法生成可消除对时恢复原排列，不把失败的随机结果留在棋盘中。

路径判断、目标发现、固定链移动、快照、恢复和回滚仍保留在 `board.js`，不依赖关卡模块。

### 5.4 `game/interaction.js`

状态新增或替换为：

```js
currentLevel
highestUnlocked
levelSeed
hintsRemaining
reshufflesRemaining
phase // loading | playing | level-complete | failed
levelSessionId
```

现有 `round` 字段和“局数”语义移除。

新增统一入口：

```js
loadLevel(levelNumber, seed)
retryCurrentLevel()
completeCurrentLevel()
```

`loadLevel()` 负责：

1. 增加 `levelSessionId`；
2. 取消旧动画、拖拽和提示定时器；
3. 清除选择、候选和未决回滚；
4. 关闭旧弹窗并清理监听；
5. 读取关卡配置并生成棋盘；
6. 重置提示和重排资源；
7. 更新顶部关卡和工具按钮；
8. 保存当前进度；
9. 渲染棋盘并进入 `playing`。

## 6. 状态流程

### 6.1 启动

1. 尝试读取存档。
2. 存档有效时加载 `currentLevel`。
3. 存档不存在、损坏或版本不兼容时加载第 1 关。
4. 第 6 关以后若合法存档缺少种子，则整份存档视为无效，不进行猜测恢复。

### 6.2 清盘过关

最后一对消除后：

1. 切换到 `level-complete`，设置 `busy = true`。
2. 计算下一关编号。
3. 更新 `highestUnlocked` 和存档中的 `currentLevel`。
4. 第 6 关以后按需生成并保存下一关种子。
5. 显示结算弹窗。
6. 点击“进入第 N+1 关”后调用 `loadLevel()`。

结算弹窗不可通过 Escape 或关闭按钮跳过。它显示：

- “第 N 关完成”；
- 当前关剩余提示和重排次数；
- 如果下一关会解锁新图案，则显示新图案和名称；
- “进入第 N+1 关”按钮。

### 6.3 重试

工具栏“重开”改为“重试”。点击后：

- 保留当前关卡编号和种子；
- 重新生成相同初始盘面；
- 恢复该关的提示与重排次数；
- 保持 `highestUnlocked`；
- 清除当前所有交互和异步状态。

首版不增加二次确认弹窗，以保持操作直接。

### 6.4 提示

- 仅在 `playing` 且不忙时响应。
- 有可消除对且次数大于零：高亮一对并扣除一次提示。
- 没有可消除对：不扣提示，进入死局流程。
- 次数为零：按钮禁用，不触发提示。

### 6.5 重排与死局

手动重排：

- 仅在 `playing`、不忙且次数大于零时响应；
- 只打乱剩余图案，保留空位；
- 成功生成至少一对可消除方块后才扣除一次重排；
- 1000 次内生成失败时恢复原排列、不扣资源，并进入“生成失败”错误态，只提供重试当前关。

自动检测到死局：

- 剩余重排大于零：显示“使用一次重排”或“重试本关”。
- 选择重排且生成成功：扣除一次并继续。
- 选择重排但生成失败：恢复原排列、不扣资源，进入“生成失败”错误态。
- 剩余重排为零：显示“本关失败”，只能重试。

不再使用当前按 `deadlockCount` 区分第一次和后续死局的规则。

## 7. UI 设计

### 7.1 顶部状态

顶部右侧由“第 N 局”改为“第 N 关”。不增加额外计分、时间或进度条，保持移动端紧凑布局。

### 7.2 工具栏

- “提示”按钮右上角显示剩余次数徽章。
- “重排”按钮右上角显示剩余次数徽章。
- “重开”文案改为“重试”。
- 次数为零时按钮使用原生 `disabled`，视觉上降低饱和度，同时保留清晰文本。
- 徽章变化通过现有 live region 或补充的屏幕阅读器文本播报。

### 7.3 弹窗

沿用现有田园街机样式和原生 `<dialog>`：

- 关卡完成弹窗：关卡编号、资源结余、新图案解锁和下一关按钮。
- 死局弹窗：使用一次重排或重试本关。
- 失败弹窗：说明资源已用尽并提供重试。

弹窗按钮、`close` 和 `cancel` 监听每次打开前都要清理；每个结果最多完成一次。

## 8. 异步安全

每次 `loadLevel()` 都递增 `levelSessionId`。以下异步回调捕获当前 ID，并在执行前检查是否仍属于当前关卡：

- 单目标选择的 220ms 延时；
- 消除表情和清空 DOM 的定时器；
- 拖拽吸附和回滚完成回调；
- 提示高亮和提示文字清理定时器；
- 结算、死局和失败弹窗回调。

旧关卡回调失效后不得：

- 修改新棋盘；
- 清除新关卡样式；
- 解除新关卡的 `busy`；
- 再次打开弹窗；
- 消耗新关卡资源。

`loadLevel()` 在生成新棋盘前统一执行：

- `moveAnimator.cancelAll()`；
- `dragInput.reset()`；
- 清理拖拽、选择、候选、未决回滚和提示计时器；
- 清理所有活动弹窗监听。

## 9. 错误与降级

### 9.1 存档

- 无存档：使用默认进度。
- JSON 损坏、字段非法或版本不兼容：丢弃整份存档，回到第 1 关。
- `localStorage` 读取或写入抛错：继续使用内存状态，并通过非阻塞提示说明本次进度无法保存。

### 9.2 随机源

- 优先使用 `crypto.getRandomValues()` 生成随机关种子。
- 若不可用，则使用一次性的 `Math.random()` 生成 32 位种子；棋盘生成本身仍使用确定性 PRNG。

### 9.3 生成失败

- `createBoard()` 或重排在 1000 次后仍找不到可消除对时返回明确失败结果，而不是静默接受死局。
- 第 1～5 关生成失败时绝不替换固定种子，直接进入“生成失败”错误态并提供重试；固定种子必须在自动化测试中预先证明可成功生成，因此该路径只作为运行时保护。
- 第 6 关以后首次生成失败时创建替代种子，先更新内存中的 `levelSeed` 并尝试写入存档，再以替代种子重试一次。
- 替代种子写入失败时，本页仍继续使用内存中的替代种子并提示进度无法保存；刷新后重新走存档校验和生成失败恢复流程。
- 替代种子仍生成失败时进入“生成失败”错误态，按钮文案为“重新生成本关”；每次点击都会生成并保存一个新种子后再次加载。
- 重排失败时恢复调用前排列且不扣重排次数，进入同一错误态，但按钮保持“重试本关”并复用当前关卡种子。

### 9.4 Reduced motion

关卡切换、弹窗和图案解锁动画遵循 `prefers-reduced-motion`。降低动态效果时直接进入最终视觉状态，不遗留动画回调或输入锁。

- 所有图案数量为偶数。

### 10.4 状态转换测试
